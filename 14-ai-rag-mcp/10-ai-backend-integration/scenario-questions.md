# Chapter 10: AI Backend Integration — Scenario Questions

1. You're building a FastAPI backend that serves streaming LLM responses to a React frontend. Users sometimes cancel requests midway through streaming. You're seeing dangling OpenAI API calls and escalating costs. Design the complete streaming architecture: the FastAPI endpoint, SSE implementation, client-side cancellation propagation, server-side connection lifecycle management, and how you handle cleanup of in-progress LLM API calls.

2. Your team is integrating GPT-4o Vision into a document processing pipeline. Documents can be PDFs with embedded images, scanned pages (images only), and mixed text/image pages. The pipeline must extract structured data (tables, key-value pairs) and work reliably on all three document types. Walk through your multi-modal processing strategy: how you handle each document type, what prompts you use, and how you structure the output for downstream systems.

3. You're migrating a production system from LangChain's legacy chain API to LCEL (LangChain Expression Language). The existing system has: a retrieval chain, a custom prompt template, a ChatOpenAI call, and a custom output parser. Walk through the migration: what each component looks like in LCEL, how you handle streaming in LCEL, and what observability improvements LCEL provides.

4. Your LLM backend is using the OpenAI API and you need it to work identically with a self-hosted Ollama instance for development/testing and a vLLM instance for production on-premise. Design the abstraction layer: what configuration approach you use, how you handle model name mapping, what API-level differences you need to accommodate, and how you test that all three backends produce equivalent results.

5. You're building a document extraction system using the Instructor library with Pydantic models. The LLM must extract: company name, founding year, revenue, and a list of products (each with name, price, and description) from unstructured press releases. Some press releases are missing information. Design the Pydantic model with appropriate Optional fields, validation, and how Instructor handles partial extraction gracefully.

6. Your async LLM application processes 1,000 documents in batch. Each document requires: one embedding call, one LLM summary call, and one classification call. Naively, this is 3,000 API calls executed sequentially. Design the async pipeline: how you structure the concurrency (asyncio.gather, semaphores, task groups), how you handle rate limits, and what the expected speedup is compared to sequential execution.

7. A team is building a LlamaIndex-based knowledge base over a codebase of 15,000 Python files. Requirements: search by semantic similarity, search by exact function name, filter by file path and modification date. Design the complete LlamaIndex pipeline: what NodeParser you would use, what metadata you would extract, how you would configure the QueryEngine, and what response synthesizer is appropriate.

8. You're implementing token counting and budget management for a production RAG system. The system needs to ensure that: (a) the system prompt + retrieved context + user query never exceed 100K tokens, (b) there's always at least 4K tokens reserved for the response, and (c) when context is too long, you truncate the least relevant chunks first (not arbitrarily). Design the token budget management logic with tiktoken and show the truncation algorithm.

9. You're building a structured output pipeline where GPT-4o must return a JSON object with a complex nested schema (loan application assessment with 15 fields, conditional requirements, and computed values). Without Instructor, your JSON parsing failure rate is 12%. Design the complete structured output strategy: when to use `response_format={"type": "json_schema"}`, when to use Instructor, and how to handle the 1-2% failure rate that persists even with the best approach.

10. Your company is evaluating LangChain vs. LlamaIndex vs. direct OpenAI SDK for a new RAG application. The application indexes 2 million technical documents, needs sub-500ms retrieval, requires custom re-ranking logic, and must run on AWS. Build a comparative evaluation across: indexing flexibility, retrieval customizability, production readiness (monitoring, error handling), learning curve, and ecosystem maturity. Give a final recommendation.
