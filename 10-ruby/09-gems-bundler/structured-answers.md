# Chapter 09: Gems & Bundler — Structured Answers

## Answer 1: How Bundler Resolves Dependencies

**Question:** "Explain the Bundler dependency resolution algorithm. What happens when there are conflicts?"

**Answer:**

```ruby
# Bundler uses a SAT (Boolean Satisfiability) solver called Molinillo
# to find a set of gem versions satisfying ALL constraints simultaneously

# Example dependency graph:
# Gemfile:
#   gem "rails", "~> 7.1"
#   gem "sidekiq", "~> 7.0"
#   gem "sentry-rails"

# Resolution process:
# 1. Fetch gemspec metadata for each gem
# 2. Build a dependency graph:
#    rails 7.1.x → needs activesupport = 7.1.x, activerecord = 7.1.x
#    sidekiq 7.0.x → needs connection_pool >= 2.3
#    sentry-rails 5.x → needs rails >= 5.0, sentry-ruby >= 5.x

# 3. Find versions satisfying all constraints:
#    activesupport must satisfy: = 7.1.x (from rails) AND >= 6.1 (from sentry?)
#    → Satisfying version: 7.1.2

# Conflict example:
# gem_a 1.0 → requires json >= 2.0
# gem_b 2.0 → requires json ~> 1.8 (i.e., < 2.0)
# → UNSATISFIABLE → Bundler raises VersionConflict

# Bundler error:
# Bundler could not find compatible versions for gem "json":
#   In Gemfile:
#     gem_a was resolved to 1.0, which depends on
#       json >= 2.0
#     gem_b was resolved to 2.0, which depends on
#       json ~> 1.8

# RESOLUTION STRATEGIES:
# 1. Update one of the conflicting gems to a version with relaxed constraints:
bundle update gem_a  # might find gem_a 1.1 which supports json >= 1.8

# 2. Pin one gem to an older compatible version:
gem "gem_a", "0.9"  # older version that works with json ~> 1.8

# 3. Verbose debugging:
bundle install --verbose
# Shows: "Resolving dependencies...", each resolution attempt

# 4. Check specific gem requirements:
bundle exec gem dependency json
# → Who requires json, what versions they need

# Resolution success → Gemfile.lock written
# Gemfile.lock = the single source of truth about exact resolved versions
```

## Answer 2: Building and Publishing a Production-Quality Gem

**Question:** "Walk me through creating, testing, and publishing a Ruby gem from scratch."

**Answer:**

```ruby
# STEP 1: Generate skeleton
bundle gem my_data_processor --coc --mit --test=rspec --ci=github

# STEP 2: Write the gemspec
# my_data_processor.gemspec
Gem::Specification.new do |spec|
  spec.name          = "my_data_processor"
  spec.version       = MyDataProcessor::VERSION
  spec.authors       = ["Your Name"]
  spec.email         = ["you@example.com"]
  spec.summary       = "Processes structured data with configurable pipelines"
  spec.homepage      = "https://github.com/yourorg/my_data_processor"
  spec.license       = "MIT"

  spec.required_ruby_version = ">= 3.0.0"

  # What files are included in the gem package
  spec.files = Dir.glob(%w[
    lib/**/*.rb
    exe/**/*
    *.md
    *.txt
    LICENSE.txt
  ])

  spec.bindir        = "exe"
  spec.executables   = spec.files.grep(%r{^exe/}) { |f| File.basename(f) }
  spec.require_paths = ["lib"]

  # Runtime deps: ONLY what's needed to use the gem
  spec.add_dependency "activesupport", ">= 6.1", "< 8"

  # Dev deps: tools for gem development
  spec.add_development_dependency "rspec",   "~> 3.12"
  spec.add_development_dependency "rubocop", "~> 1.57"
end

# STEP 3: Version management
# lib/my_data_processor/version.rb
module MyDataProcessor
  VERSION = "1.0.0"
  # Follow SemVer: MAJOR.MINOR.PATCH
  # - PATCH for bug fixes (1.0.0 → 1.0.1)
  # - MINOR for new features, backward-compatible (1.0.0 → 1.1.0)
  # - MAJOR for breaking changes (1.0.0 → 2.0.0)
end

# STEP 4: Write the code
# lib/my_data_processor.rb
require "my_data_processor/version"
require "my_data_processor/pipeline"
require "my_data_processor/processor"

module MyDataProcessor
  def self.process(data, **options)
    Pipeline.new(**options).run(data)
  end
end

# STEP 5: Write comprehensive tests
# spec/my_data_processor_spec.rb
RSpec.describe MyDataProcessor do
  describe ".process" do
    it "processes data through the pipeline" do
      result = MyDataProcessor.process([1, 2, 3], transform: :double)
      expect(result).to eq([2, 4, 6])
    end
  end
end

# STEP 6: Add CI (GitHub Actions auto-generated by bundle gem --ci=github)
# .github/workflows/main.yml already configured

# STEP 7: Publish
# First time: create account on rubygems.org
gem signin  # stores credentials in ~/.gem/credentials

# Build
gem build my_data_processor.gemspec
# → my_data_processor-1.0.0.gem

# Verify the gem contents before publishing
gem contents my_data_processor-1.0.0.gem

# Publish
gem push my_data_processor-1.0.0.gem

# STEP 8: Automate releases with Rake tasks (already in skeleton)
rake release
# Bumps version, builds gem, creates git tag, pushes to rubygems.org
```

## Answer 3: Understanding the Load Path and `require`

**Question:** "How does Ruby decide what file to load when you `require 'some_gem'`? How does Bundler affect this?"

**Answer:**

```ruby
# Without Bundler:
$LOAD_PATH
# => [
#   "/home/user/.rbenv/versions/3.2.0/lib/ruby/gems/3.2.0/gems/rails-7.1.2/lib",
#   "/home/user/.rbenv/versions/3.2.0/lib/ruby/gems/3.2.0/gems/activesupport-7.1.2/lib",
#   "/home/user/.rbenv/versions/3.2.0/lib/ruby/3.2.0",
#   "."  # current directory
# ]

# When you call: require 'rails'
# Ruby searches $LOAD_PATH in order for: rails.rb or rails.so
# First match wins

# With Bundler (bundle exec or Bundler.setup):
require 'bundler/setup'

$LOAD_PATH
# → Bundler modifies $LOAD_PATH to include ONLY the locked gem versions
# → System gems are NOT in the path (unless they match Gemfile.lock versions)

# Example:
# System has: rails 7.0.4 AND rails 7.1.2
# Gemfile.lock says: rails 7.1.2
# After Bundler.setup:
# $LOAD_PATH includes: .../gems/rails-7.1.2/lib  (correct version)
# $LOAD_PATH excludes: .../gems/rails-7.0.4/lib  (wrong version)

# How Bundler.setup works:
require 'bundler'
Bundler.setup(:default, :development)  # adds groups to $LOAD_PATH

# In Rails:
# config/boot.rb:
require 'bundler/setup'  # modifies $LOAD_PATH
Bundler.require(*Rails.groups)  # auto-requires all gems in active groups

# Require path aliases:
gem "redis-rb", require: "redis"
# → When Bundler.require is called, it requires 'redis' not 'redis-rb'

# Debugging $LOAD_PATH issues:
puts $LOAD_PATH.grep(/nokogiri/).inspect
# See exactly which nokogiri version is in the path
```

## Answer 4: Private Gems in Enterprise Environments

**Question:** "How do you set up a private gem server for internal gems in a company?"

**Answer:**

```bash
# Option 1: Gemfury (managed service, simplest)
# Sign up at gemfury.com

# Push a gem:
gem push my_private_gem-1.0.0.gem --host https://gem.fury.io/mycompany/

# Configure team access:
# .bundle/config or Bundler config per developer:
bundle config set --global https://gem.fury.io/mycompany/ "FURY_TOKEN:x"

# Gemfile:
source "https://gem.fury.io/mycompany/" do
  gem "company_auth"
  gem "shared_models"
end

# CI configuration (GitHub Actions):
# secrets: GEMFURY_TOKEN
# env: BUNDLE_GEM__FURY__IO__MYCOMPANY__: "${{ secrets.GEMFURY_TOKEN }}:x"
```

```ruby
# Option 2: GitHub Packages (for GitHub-hosted orgs)
# Gemfile:
source "https://rubygems.pkg.github.com/myorg" do
  gem "private_gem"
end

# Authentication:
bundle config set --global https://rubygems.pkg.github.com/myorg \
  "ghp_your_github_token:x"

# Pushing:
gem push --key github \
  --host https://rubygems.pkg.github.com/myorg \
  private_gem-1.0.0.gem
```

```bash
# Option 3: geminabox (self-hosted, lightweight)
# Server setup:
gem install geminabox
mkdir /var/gems
geminabox /var/gems  # starts a gem server on port 9292

# Client Gemfile:
source "http://gems.internal.company.com:9292"
gem "internal_lib"

# Pushing:
gem inabox my_internal_lib-1.0.0.gem
```

## Answer 5: Gemfile Best Practices for a Team

**Question:** "What are your Gemfile best practices for a team of 5 developers on a Rails API?"

**Answer:**

```ruby
# frozen_string_literal: true

source "https://rubygems.org"

ruby file: ".ruby-version"  # enforce Ruby version from .ruby-version file

# Web framework
gem "rails", "~> 7.1"

# Database
gem "pg", "~> 1.5"

# Authentication & authorization
gem "devise", "~> 4.9"

# Background jobs

# HTTP

# Serialization

# Monitoring
gem "sentry-rails", "~> 5.14"
gem "sentry-sidekiq", "~> 5.14"

# Performance
gem "bootsnap", require: false  # startup speed

group :development, :test do
  gem "rspec-rails", "~> 6.0"
  gem "factory_bot_rails", "~> 6.4"
  gem "faker", "~> 3.2"        # for factories
  gem "pry-rails"              # debugging
  gem "database_cleaner-active_record"
end

group :development do
  gem "annotate"               # model annotations
  gem "bullet"                 # N+1 detection
  gem "letter_opener"          # preview emails in browser
  gem "rubocop", require: false
  gem "rubocop-rails", require: false
  gem "rubocop-rspec", require: false
end

group :test do
  gem "webmock"               # HTTP request stubbing
  gem "vcr", "~> 6.2"         # cassette recording
  gem "simplecov", require: false  # code coverage
  gem "timecop"               # time manipulation
end

# Platform-specific gems:
gem "tzinfo-data", platforms: %i[mingw mswin x64_mingw jruby]
gem "wdm", ">= 0.1.0", platforms: %i[mingw mswin x64_mingw]

# PRACTICES:
# 1. Use ~> for all gems with known minor-breaking tendencies
# 2. No version constraint for your own internal gems (update frequently)
# 3. Group properly: expensive gems in test/development groups
# 4. Document unusual version pins with inline comments
# 5. Use require: false for gems that are conditionally needed
# 6. Commit Gemfile.lock
# 7. Use `bundle audit` in CI
# 8. Keep Gemfile alphabetically sorted within groups (easier diffs)
```

## Answer 6: Diagnosing and Fixing `bundle install` Failures

**Question:** "Walk me through diagnosing common `bundle install` failures."

**Answer:**

```bash
# FAILURE TYPE 1: Version conflict
# Error: Bundler could not find compatible versions for gem "X"

# Fix: update conflicting gems together
bundle update gem_a gem_b  # resolve together

# FAILURE TYPE 2: Native extension compilation failure
# Error: Gem::Ext::BuildError: ERROR, Failed to build gem native extension
# Building nokogiri (1.15.4) with native extensions...
# make failed, exit code 2

# Fix: install system dependencies
# macOS:
brew install libxml2 libxslt

# Ubuntu/Debian:
apt-get install libxml2-dev libxslt-dev

# Then retry:
bundle install

# Better: use pre-compiled binary gems (modern nokogiri supports this)
bundle config set --local force_ruby_platform false
bundle install  # will use pre-compiled gem for your platform

# FAILURE TYPE 3: Ruby version mismatch
# Error: Your Ruby version is 2.7.5, but your Gemfile specified 3.0.0

# Fix:
rbenv install 3.0.0
rbenv local 3.0.0
bundle install

# FAILURE TYPE 4: Network failure
# Error: Retrying source fetch due to error (2/4): Bundler::Fetcher::FallbackError

# Fix:
bundle install --retry=5  # increase retries
# or use a mirror:
bundle config set --global mirror.https://rubygems.org https://rubygems.org.cn  # China mirror example

# FAILURE TYPE 5: Frozen bundle out of date (CI)
# Error: The `--frozen` flag is set and the Gemfile.lock is out of date.

# Fix: update Gemfile.lock locally and commit
bundle install
git add Gemfile.lock && git commit -m "Update Gemfile.lock"
git push

# FAILURE TYPE 6: Missing gemspec file
# Error: There are no gemspecs at /path/to/gem

# Fix: verify you're in the right directory and gemspec exists
ls *.gemspec  # should show exactly one .gemspec file

# DEBUGGING ANY FAILURE:
bundle install --verbose 2>&1 | tee /tmp/bundle_output.txt
# Full verbose output helps identify exactly where resolution fails
```

## Answer 7: Gem Lifecycle and Maintenance

**Question:** "Once a gem is published, what ongoing maintenance responsibilities does a gem author have?"

**Answer:**

```ruby
# ONGOING MAINTENANCE TASKS:

# 1. Security monitoring
# Subscribe to ruby-advisory-db PR notifications:
# https://github.com/rubysec/ruby-advisory-db
# When CVEs are discovered, publish patches immediately

# 2. Ruby compatibility
# Test against new Ruby versions as they release
# (typically May for previews, December for stable)

# .github/workflows/test.yml:
strategy:
  matrix:
    ruby: ["3.1", "3.2", "3.3", "head"]  # test against current + next

# 3. Dependency updates
# Check for outdated dependencies monthly:
bundle outdated

# 4. Respond to issues
# Keep GitHub Issues list manageable
# Mark old issues as "needs-triage", "wontfix", or "help-wanted"

# 5. Changelog maintenance
# Keep CHANGELOG.md updated with each release:
## [1.3.0] - 2026-03-17
### Added
- Support for Ruby 3.3
- New `pipeline_timeout` option

### Fixed
- Memory leak in Pipeline#run when exception occurs

### Changed
- Minimum Ruby version is now 3.1

## [1.2.1] - 2026-01-10
### Security
- Fixed CVE-2026-XXXX: input not sanitized before eval

# 6. Version policy
# Communicate your versioning policy in README:
# - MAJOR: breaking changes
# - MINOR: new features, backward compatible
# - PATCH: bug fixes only

# 7. Yanking a gem (emergency: security vulnerability)
gem yank my_gem -v 1.2.3  # removes from rubygems.org
# Users get warning; Gemfile.lock still references yanked version
# → publish fixed version as 1.2.4 immediately

# 8. Deprecation workflow
# Before removing a feature:
def old_method
  warn "[DEPRECATION] `old_method` is deprecated. Use `new_method` instead. (called from #{caller[0]})"
  new_method
end
# Keep deprecated methods for at least ONE MINOR version
```

## Answer 8: Configuring Bundler for Production Deployments

**Question:** "How do you configure Bundler differently for development vs production vs Docker?"

**Answer:**

```bash
# DEVELOPMENT:
# .bundle/config (developer's local config, NOT committed):
bundle config set --local path 'vendor/bundle'  # local install
bundle config set --local without ''            # install all groups
bundle install --jobs=4

# PRODUCTION (Heroku, VM):
# Heroku auto-configures:
# BUNDLE_WITHOUT: "development:test"
# BUNDLE_DEPLOYMENT: "1"  (equivalent to --deployment)
# OR manually in deploy script:
bundle config set --global without 'development test'
bundle config set --global deployment true  # requires Gemfile.lock committed
bundle install --jobs=4 --retry=3

# CI (GitHub Actions):
- name: Configure Bundler
  run: |
    bundle config set --local without 'production'
    bundle config set --local path 'vendor/bundle'
    bundle config set --local frozen true  # fail if lock is out of date

- name: Cache gems
  uses: actions/cache@v3
  with:
    path: vendor/bundle
    key: gems-${{ hashFiles('Gemfile.lock') }}

- name: Install gems
  run: bundle install --jobs=4 --retry=3

# DOCKER:
# Dockerfile:
FROM ruby:3.2-slim

WORKDIR /app

# Install gems first (separate layer = cacheable)
COPY Gemfile Gemfile.lock ./
RUN bundle config set --without 'development test' && \
    bundle config set --jobs 4 && \
    bundle install --frozen

# Copy the rest of the code (doesn't bust gem cache)
COPY . .

# Multi-stage Docker (gems in builder, only production in final):
FROM ruby:3.2-slim AS builder
WORKDIR /app
COPY Gemfile Gemfile.lock ./
RUN bundle install --without development test --frozen

FROM ruby:3.2-slim AS production
WORKDIR /app
COPY --from=builder /usr/local/bundle /usr/local/bundle
COPY . .
CMD ["bundle", "exec", "puma"]
```
