**Trap 1: "You said Rails auto-escapes ERB output. Is `<%= @user.bio.html_safe %>` safe if you trust the user?"**

Never trust user input as HTML-safe. `html_safe` marks ANY string as safe, bypassing all escaping. Even "trusted" users can be victims of stored XSS if their input is later injected into another user's browser.

```ruby
# Scenario: Alice edits her bio to contain XSS payload
# Your admin views Alice's profile
# The <script> in Alice's bio executes as admin!

# "I trust Alice" is not sufficient — consider:
# 1. Alice's account could be compromised
# 2. You might reuse this pattern for untrusted users later
# 3. The code pattern spreads through copy-paste

# NEVER:
@user.bio.html_safe
raw(@user.bio)
"<b>#{@user.name}</b>".html_safe  # interpolation in html_safe string is SAFE
                                    # because @user.name is escaped during interpolation

# SAFE ways to include HTML:
# 1. Let Rails auto-escape (default) — shows literal tags as text
<%= @user.bio %>

# 2. Sanitize — strips disallowed tags, keeps allowed
<%= sanitize @user.bio %>

# 3. ActionText (handles rich text safely)
# 4. Markdown with sanitization
<%= sanitize Redcarpet::Markdown.new(Redcarpet::Render::HTML.new).render(@user.bio) %>
```

**Trap 2: "You use `<% cache @article do %>`. What is the cache key? What happens if you add a column to articles?"**

```ruby
# Cache key generated by AR's cache_key_with_version:
# "articles/42-20240115120000000000000"
#  model_class / id - updated_at_nanoseconds

# Adding a column doesn't change the cache key unless:
# - The record is saved (updated_at changes)
# - You include the schema version in the key

# GOTCHA: If you add a `featured` boolean column and update articles via migration:
# update_all(featured: false)  → does NOT change updated_at → stale cache
# The cache still shows the old view (without the new featured badge)

# Solution: always use update! or touch for display-relevant changes:
Article.find_each { |a| a.touch }  # Update updated_at → invalidates all caches

# Better: include schema digest in cache key:
<% cache [@article, "v3"] do %>  <%# Bump version string on template changes %>

# Or use cache_key_with_version (includes a digest of the template):
<% cache [cache_prefix, @article] do %>
<%# Use Rails.cache_digest: digest of the template file is included %>
```

**Trap 3: "You said Turbo Drive intercepts all link clicks. What happens with external links, downloads, and links that need to stay HTML-only?"**

Turbo Drive intercepts same-origin links only. External links still do full navigation. But there are cases within your app where Turbo causes issues:

```erb
<%# These cause problems with Turbo Drive: %>

<%# 1. Links that trigger browser download %>
<%= link_to "Download PDF", invoice_pdf_path(@invoice), data: { turbo: false } %>
<%# Without data-turbo="false", Turbo fetches the PDF as HTML and replaces body %>

<%# 2. Links that should trigger a full page reload (logout, changing locales) %>
<%= link_to "Logout", logout_path, data: { turbo: false } %>
<%# Session changes may need a full page to update properly %>

<%# 3. File upload forms with multipart: true %>
<%= form_with model: @avatar, html: { enctype: "multipart/form-data" },
              data: { turbo: false } do |f| %>

<%# 4. Pages with JavaScript that initializes on DOMContentLoaded %>
<%# Turbo Drive doesn't fire DOMContentLoaded on navigation %>
<%# Fix: use turbo:load event instead %>
document.addEventListener("turbo:load", function() {
  // Initialize charts, maps, etc.
})
```

**Trap 4: "You add `<% cache @article do %>` and Russian doll caches for comments. Updating a comment invalidates the article cache too. But what about updating the AUTHOR of a comment?"**

```ruby
# Scenario:
# - Article caches comment.author.name
# - Comment author changes their name
# - comment.touch fires → article.touch fires (via belongs_to :article, touch: true)
# CORRECT: article cache invalidates

# But: what if author is associated through user, not cached directly?
# Comment belongs_to :author (User)
# If User changes their name: User is saved → updated_at changes
# Does this touch Comment? NO — unless Comment belongs_to :author, touch: true!
# Does this touch Article? NO — unless touch: true chain is complete

# Fix: ensure the full touch chain exists
class User < ApplicationRecord
  has_many :comments
  after_save :touch_comments, if: :saved_change_to_name?
  def touch_comments
    comments.update_all(updated_at: Time.current)
    # This triggers article touch via belongs_to :article, touch: true on Comment
  end
end

# Or: use a cache key that includes the author's name explicitly
<% cache [comment, comment.author.name] do %>
```

**Trap 5: "Is `link_to 'Delete', article, method: :delete` safe in Rails 7?"**

No — this approach is DEPRECATED in Rails 7. `link_to` with `method: :delete` used to work via the `rails-ujs` JavaScript library which intercept clicks and submit a form. In Rails 7, `rails-ujs` is no longer included by default (replaced by Turbo).

```erb
<%# Rails 6 (works with rails-ujs): %>
<%= link_to "Delete", article_path(@article), method: :delete %>
<%# Generated: <a href="/articles/1" data-method="delete">Delete</a> %>
<%# rails-ujs JS intercepts click, creates form, submits DELETE %>

<%# Rails 7 (rails-ujs removed): %>
<%# The above renders an <a> tag that does a GET request! Article is NOT deleted! %>
<%# Silent bug — no error, just wrong HTTP method used %>

<%# Correct Rails 7 approach: %>
<%= button_to "Delete", article_path(@article), method: :delete,
              form: { data: { turbo_confirm: "Are you sure?" } } %>
<%# Generates a <form method="post"> with _method=delete + CSRF token %>
<%# Turbo (not rails-ujs) handles confirmation %>

<%# If you must use link_to: %>
<%= link_to "Delete", article_path(@article),
            data: { turbo_method: :delete, turbo_confirm: "Are you sure?" } %>
<%# data-turbo-method="delete" is handled by Turbo Drive %>
```

**Trap 6: "Partials use instance variables by default. Why is this a problem and how do you fix it?"**

```erb
<%# BAD: partial implicitly uses instance variables from controller %>
<%# app/views/articles/_article.html.erb %>
<div>
  <h2><%= @article.title %></h2>  <%# @article from controller %>
  <p><%= @article.author.name %></p>
</div>

<%# Problems: %>
<%# 1. Partial is tightly coupled to controller — hard to reuse %>
<%# 2. Testing is hard (must mock controller instance variables) %>
<%# 3. In a collection render, @article is wrong for all but the last item %>
<%# 4. When rendered in different context, @article may be nil or wrong object %>

<%# GOOD: pass locals explicitly %>
<%# app/views/articles/_article.html.erb %>
<div>
  <h2><%= article.title %></h2>  <%# local variable — explicit and safe %>
  <p><%= article.author.name %></p>
</div>

<%# Render with locals: %>
<%= render partial: "article", locals: { article: @article } %>
<%# Or via implicit local from collection (same name as partial): %>
<%= render @article %>  <%# local 'article' set automatically %>
<%= render @articles %> <%# local 'article' set for each item %>
```

**Trap 7: "form_with uses `local: false` by default in Rails 5/6. Is this still true in Rails 7?"**

This changed between versions:

```ruby
# Rails 5.1 - 6.x: form_with defaults to data-remote="true" (AJAX form)
<%= form_with model: @article %>
# Generates: <form data-remote="true" ...>
# Submitted via AJAX (requires rails-ujs to handle)

# Rails 6.1+: form_with defaults changed to local: true (standard form submission)
# The old AJAX behavior required explicit: data: { remote: true }

# Rails 7: Turbo handles all form submissions (via fetch/AJAX) automatically
# The local: true vs local: false distinction is no longer as relevant
# Turbo intercepts ALL form submissions unless data-turbo="false"

# data-turbo="false" disables Turbo for that form (standard full-page form submission):
<%= form_with model: @article, data: { turbo: false } do |f| %>
# Disables Turbo — form submits as traditional full-page form
```

**Trap 8: "You render a partial in a loop: `@articles.each { |a| render 'article', locals: { article: a } }`. What's the performance issue?"**

```ruby
# Each render call in a loop:
# 1. Finds the partial template file
# 2. Compiles the ERB template
# 3. Creates a new renderer context
# Repeated N times for N records

# The fix: collection rendering (single partial lookup, batched rendering)
render partial: "article", collection: @articles
# OR:
render @articles

# Performance difference (1000 items):
# Loop render:       ~500ms template overhead
# Collection render: ~50ms (template found/compiled once, applied N times)

# The article_counter local variable is provided automatically:
# In _article.html.erb: article_counter gives 0-based index
# article_iteration.index, article_iteration.first?, article_iteration.last?

# Cache + collection (Russian doll optimized):
render partial: "article", collection: @articles, cached: true
# Each item's cache is checked, only misses are rendered
```

**Trap 9: "You use `number_to_currency(@price)`. A user from Germany sees `$1.234,56` instead of `1.234,56 €`. How do you fix the locale-specific formatting?"**

```ruby
# number_to_currency uses the CURRENT I18n.locale settings
# If locale is :en, it uses US dollar format by default

# Fix 1: Set locale per user in ApplicationController:
around_action :set_locale

def set_locale
  I18n.with_locale(current_user&.locale || I18n.default_locale) { yield }
end

# config/locales/de.yml:
de:
  number:
    currency:
      format:
        unit: "€"
        precision: 2
        separator: ","
        delimiter: "."
        format: "%n %u"  # "1.234,56 €" (number first, unit last)

# Fix 2: Explicit parameters (always works regardless of locale):
number_to_currency(@price,
  unit: current_user.currency_symbol,
  separator: current_user.decimal_separator,
  delimiter: current_user.thousands_delimiter
)

# Fix 3: Use Money gem for correct currency handling:
# @product.price is a Money object
@product.price.format(locale: I18n.locale)
```

**Trap 10: "How does `content_for` work with multiple layouts? If a template calls `content_for :sidebar`, but the layout doesn't have `yield :sidebar`, what happens?"**

The content is silently ignored — `content_for` stores content in a hash associated with the view context. If `yield :sidebar` is never called in the layout, that content is never rendered and is simply discarded.

```erb
<%# Template sets sidebar content %>
<% content_for :sidebar do %>
  <p>Related articles</p>
<% end %>

<%# Layout A: has sidebar %>
<div class="sidebar"><%= yield :sidebar %></div>
<%# Renders "Related articles" %>

<%# Layout B: no sidebar (e.g., admin layout) %>
<%# yield :sidebar is never called %>
<%# The "Related articles" content is prepared but never output %>
<%# No error, no warning — silent discard %>

# Check if content was provided:
if content_for?(:sidebar)
  # Layout dynamically adapts:
  content_tag(:div, yield(:sidebar), class: "sidebar")
end
```
