## ORM Fundamentals

**Q1. What is the Active Record pattern and how does Rails' implementation differ from Martin Fowler's original definition?**

Martin Fowler's pattern: an object that wraps a database row and encapsulates database access logic. Rails' implementation follows this closely, but adds the Query Interface (scopes, relations) which are more DataMapper-like. Rails AR objects carry both business logic AND query building.

---

**Q2. What is `schema.rb` vs `structure.sql`? When would you choose each?**

- `schema.rb`: Ruby DSL representation of the schema, database-agnostic. Generated by `db:schema:dump`. Used by `db:schema:load` for fresh test databases.
- `structure.sql`: Raw SQL DDL from the database (`pg_dump` for PostgreSQL). Preserves database-specific features: custom types, stored procedures, views, check constraints, partial indexes, triggers.

Choose `structure.sql` when using PostgreSQL-specific features. Set via `config.active_record.schema_format = :sql`.

---

**Q3. What is a Rails migration? What does `rails db:migrate` do vs `rails db:schema:load`?**

- `db:migrate`: Runs pending migrations sequentially, updating `schema_migrations` table
- `db:schema:load`: Loads `schema.rb` directly, bypassing all migrations. MUCH faster for setting up a fresh database. Does NOT run migrations.

Never run `db:schema:load` in production on a database with data — it drops and recreates tables.

```ruby
class User < ApplicationRecord

  validates :name, presence: true

  validates :birth_date, :not_in_future

  end
end
```

**Q5. What is the difference between `save`, `save!`, `update`, `update!`, `update_attribute`, and `update_columns`?**

```ruby
# save: runs validations + callbacks, returns true/false
user.name = "Alice"
user.save  # false if invalid

# save!: same but raises ActiveRecord::RecordInvalid on failure
user.save!  # raises on invalid

# update: mass-assigns + save, runs validations + callbacks
user.update(name: "Alice")  # false if invalid

# update!: same but raises on failure
user.update!(name: "Alice")  # raises on invalid

# update_attribute: updates ONE column, SKIPS validations, runs callbacks
user.update_attribute(:name, "Alice")  # bypasses validates :name

# update_columns: updates multiple columns, SKIPS validations AND callbacks
user.update_columns(name: "Alice", email: "a@b.com")  # raw SQL UPDATE, no callbacks

# update_column: updates ONE column, SKIPS validations AND callbacks
user.update_column(:name, "Alice")  # raw SQL UPDATE, no callbacks
```

**Q7. What is the difference between `after_save` and `after_commit`? When does each fire?**

- `after_save`: fires after the SQL INSERT/UPDATE runs but BEFORE the transaction commits. If the transaction is rolled back, `after_save` has already run but the DB change is undone.
- `after_commit`: fires only after the transaction is fully committed to the database. Safe for triggering external side effects (sending emails, enqueuing Sidekiq jobs, invalidating caches).

```ruby
# DANGEROUS: email sent even if transaction rolls back
after_save :send_welcome_email

# SAFE: email only sent if the record is actually persisted
after_commit :send_welcome_email, on: :create
```

**Q8. Explain the 6 types of ActiveRecord associations. Give the migration and model code for each.**

```ruby

class Comment < ApplicationRecord
  belongs_to :post  # requires post_id column on comments
end

# has_many — foreign key on OTHER table
class Post < ApplicationRecord
  has_many :comments  # comments.post_id
end

class User < ApplicationRecord
  has_one :profile  # profiles.user_id
end

class Doctor < ApplicationRecord
  has_many :appointments
  has_many :patients, through: :appointments
end
class Appointment < ApplicationRecord
  belongs_to :doctor
  belongs_to :patient
end

# has_one :through
class Account < ApplicationRecord

  has_one :history, through: :user
end

# has_and_belongs_to_many — join table WITHOUT a model
class Article < ApplicationRecord
  has_and_belongs_to_many :tags  # requires articles_tags join table
end

# Polymorphic — single association across multiple models
class Picture < ApplicationRecord
  belongs_to :imageable, polymorphic: true
  # requires imageable_id and imageable_type columns
end
class Employee < ApplicationRecord
  has_many :pictures, as: :imageable
end
class Product < ApplicationRecord
  has_many :pictures, as: :imageable
end
```

**Q9. What is an N+1 query? How do you detect it and fix it?**

N+1: loading N records then running one query per record to load an association.

```ruby
# N+1 BUG: 1 query for posts + N queries for authors
posts = Post.all
posts.each { |post| puts post.author.name }
# SQL: SELECT * FROM posts → then for each post: SELECT * FROM users WHERE id=?

# Fix 1: includes (smart — uses preload or eager_load)
posts = Post.includes(:author).all
posts.each { |post| puts post.author.name }
# SQL: SELECT * FROM posts; SELECT * FROM users WHERE id IN (1,2,3,...)

# Fix 2: preload (always separate query)
posts = Post.preload(:author)

# Fix 3: eager_load (always JOIN)
posts = Post.eager_load(:author)
# SQL: SELECT posts.*, users.* FROM posts LEFT OUTER JOIN users ON ...

# Detection: Bullet gem in development
# gem 'bullet', group: :development
config.after_initialize do
  Bullet.enable = true
  Bullet.rails_logger = true
  Bullet.add_footer = true
end
```

**Q10. What is the difference between `includes`, `preload`, and `eager_load`?**

| Method | SQL Strategy | Notes |
|--------|-------------|-------|
| `preload` | Always separate queries (IN clause) | Cannot use association columns in WHERE |
| `eager_load` | Always LEFT OUTER JOIN | Can filter/sort by association columns in same query |
| `includes` | Decides: preload by default, switches to eager_load if WHERE/ORDER references association table | Best default choice |

```ruby
# Use eager_load when filtering by association:
Post.eager_load(:comments).where(comments: { approved: true })
# SQL: SELECT * FROM posts LEFT OUTER JOIN comments ON ... WHERE comments.approved = true

# Use preload when you DON'T need to filter (avoids large JOIN result sets):
Post.preload(:comments, :tags, :author)
# 3 separate queries — faster when each association has many records
```

**Q11. What is a default_scope? Why is it considered dangerous?**

```ruby
# Definition:
class Post < ApplicationRecord
  default_scope { where(published: true) }
end

# DANGER 1: applies to ALL queries including admin operations
Post.all                    # WHERE published = true (surprising!)
Post.where(user_id: 1)     # WHERE user_id = 1 AND published = true

# DANGER 2: unscope is easy to forget
Post.unscoped.all          # Must explicitly remove default scope

# DANGER 3: affects new record defaults
Post.new.published          # => true (attribute set from default_scope)

# DANGER 4: breaks joins
User.joins(:posts)          # Applies posts' default_scope to JOIN — surprising WHERE clause

# Better alternative: named scopes
class Post < ApplicationRecord
  scope :published, -> { where(published: true) }
  # Use Post.published explicitly where needed
end
```

**Q12. What is optimistic vs pessimistic locking? When do you use each?**

```ruby
# OPTIMISTIC LOCKING
# Add lock_version integer column to migration:
# add_column :accounts, :lock_version, :integer, default: 0

class Account < ApplicationRecord
  # lock_version enabled automatically when column exists
end

account1 = Account.find(1)
account2 = Account.find(1)

account1.update!(balance: 100)  # lock_version becomes 1
account2.update!(balance: 200)  # RAISES ActiveRecord::StaleObjectError
# (account2 still has lock_version: 0, but DB now has 1)

# Use when: low contention, reads >> writes, e.g. editing a document
# Benefit: no DB lock held, reads never block

# PESSIMISTIC LOCKING
Account.transaction do
  account = Account.lock.find(1)  # SELECT ... FOR UPDATE
  # DB holds a row lock — no other process can update this row
  account.update!(balance: account.balance - 50)
end  # lock released on commit

# Use when: high contention, e.g. decrementing inventory counts
# Benefit: guaranteed consistency; Drawback: can cause deadlocks
```

**Q13. What is Single Table Inheritance (STI) in ActiveRecord? What are its tradeoffs?**

```ruby
# Migration:
add_column :vehicles, :type, :string

class Vehicle < ApplicationRecord; end
class Car < Vehicle
  def honk = "Beep!"
end

  def honk = "HOOOONK!"

class Motorcycle < Vehicle; end

# ALL stored in vehicles table with type column:
Car.create!(make: "Toyota")     # type: "Car"
Truck.create!(make: "Ford")     # type: "Truck"

# Pros:
# - Simple — no JOINs for polymorphic queries
# - Shared columns natural
# Cons:
# - Table has nullable columns (Car doesn't use payload_capacity from Truck)
# - All types queried from one table — can cause table lock contention
# - Hard to enforce per-subclass column constraints
# - Not Null constraints can't apply to subclass-specific columns
```

**Q14. What is `counter_cache`? How does it work? What are its gotchas?**

```ruby
class Comment < ApplicationRecord
  belongs_to :post, counter_cache: true
  # Requires posts.comments_count integer column, default: 0
end

# Rails maintains comments_count automatically via callbacks:
# On create: UPDATE posts SET comments_count = comments_count + 1
# On destroy: UPDATE posts SET comments_count = comments_count - 1

# Access:
post.comments.size   # Uses comments_count (no SQL!)
post.comments.count  # Hits the DB with COUNT (ignores counter_cache)
post.comments.length # Loads all comments into memory

# Gotcha 1: bulk operations bypass callbacks
Comment.where(post_id: 1).delete_all  # Does NOT decrement counter_cache!
# Fix: use destroy_all (fires callbacks) or manually recalculate:
Post.reset_counters(post.id, :comments)

# Gotcha 2: custom counter cache name:
belongs_to :post, counter_cache: :total_comments_count
```

**Q15. What is dirty tracking in ActiveRecord? Explain the `changed?`, `changes`, `previous_changes` methods.**

```ruby
user = User.find(1)  # name: "Alice"
user.name = "Bob"

user.changed?              # => true
user.changed               # => ["name"]
user.name_changed?         # => true

user.previous_changes      # => { "name" => ["Alice", "Bob"] } (what changed in last save)

# Use in callbacks:

  if name_changed? && !new_record?
    NameChangeNotifier.notify(self, from: name_was, to: name)
  end
end
```

**Q17. What is `enum` in ActiveRecord? Show common usage and pitfalls.**

```ruby
class Order < ApplicationRecord
  enum :status, {
    pending:    0,
    processing: 1,
    shipped:    2,
    delivered:  3,
    cancelled:  4
  }
  # Generates: Order.pending, Order.shipped (scopes)
  # order.pending?, order.shipped? (predicates)
  # order.pending!, order.shipped! (bang mutators)
end

order = Order.create!(status: :pending)
order.pending?     # => true
order.processing!  # UPDATE orders SET status = 1
Order.shipped      # WHERE status = 2

# PITFALL 1: Integer values are hard to read in DB
# Better: use string enum with prefix
enum :status, { pending: "pending", shipped: "shipped" }, prefix: :status
# Adds: order.status_pending?

# PITFALL 2: Never change integer values — it corrupts data
# If you add a status, always append new integers, never reorder

# PITFALL 3: enum raises ArgumentError for invalid values
order.status = "invalid"  # => ArgumentError: 'invalid' is not a valid status
# Protection: rescue or validate before assignment
```

**Q19. What are ActiveRecord transactions? What is the `requires_new: true` option?**

```ruby
# Basic transaction — all-or-nothing
ActiveRecord::Base.transaction do
  account_from.update!(balance: account_from.balance - 100)
  account_to.update!(balance: account_to.balance + 100)
  # If any raises, BOTH updates are rolled back
end

# Nested transactions (savepoints with requires_new):
Account.transaction do
  account.update!(balance: 100)  # outer transaction

  Account.transaction(requires_new: true) do
    audit_log.create!(...)  # inner savepoint
    raise ActiveRecord::Rollback  # only rolls back the inner savepoint
  end
  # outer transaction continues
end

# Without requires_new: true, raising ActiveRecord::Rollback in nested
# transaction is silently ignored (it only signals the outer transaction)

# After_commit vs after_save timing:
class Order < ApplicationRecord
  after_save :enqueue_job  # WRONG: fires inside transaction, job may run before commit
  after_commit :enqueue_job  # CORRECT: fires after DB commit
end
```

**Q20. What are ActiveRecord scopes? How do default_scope and named scopes differ?**

```ruby
class Article < ApplicationRecord
  # Named scope — explicitly called
  scope :published, -> { where(published: true) }
  scope :recent,    -> { order(created_at: :desc).limit(10) }
  scope :by_author, ->(author) { where(author: author) }
  scope :search,    ->(query) { where("title ILIKE ?", "%#{query}%") }

  # Scopes are chainable:
  Article.published.recent.by_author("Alice")
  # SQL: WHERE published = true ORDER BY created_at DESC LIMIT 10 AND author = 'Alice'

  # Scope with condition (returns all if blank):
  scope :search, ->(query) {
    where("title ILIKE ?", "%#{query}%") if query.present?
  }
end

# Class methods can replace scopes and are more flexible:
def self.older_than(days)
  where("created_at < ?", days.days.ago)
end
# Use class method when scope has complex conditional logic
```

**Q21. What is `touch` in ActiveRecord associations? What problem does it solve?**

```ruby
class Comment < ApplicationRecord
  belongs_to :post, touch: true
  # When comment is saved/destroyed, post.updated_at is updated automatically
end

class Post < ApplicationRecord
  belongs_to :user, touch: true

# Why: cache invalidation
# fragment caching on posts uses cache_key_with_version (includes updated_at)
# When a comment changes, post's updated_at changes → cache key changes → cache miss

# Manual touch:
post.touch                          # updates updated_at
post.touch(:published_at)           # updates a custom timestamp column
```

**Q22. Explain the `includes` vs `joins` distinction. When does `includes` use a JOIN?**

```ruby
# includes — loads associations into memory (for use in Ruby)
Post.includes(:comments)
# Purpose: prevent N+1 when iterating through posts and accessing comments
# Strategy: usually runs 2 queries (preload strategy)

# joins — adds SQL JOIN to filter/sort (no association loading)
Post.joins(:comments).where(comments: { approved: true })
# Purpose: filter posts BASED ON comments; doesn't load comments into Post objects
# Result: post.comments still triggers a new query (N+1!)

# COMBINE them when you need both filtering AND loading:
Post.joins(:comments)
    .where(comments: { approved: true })
    .includes(:comments)
# Or use eager_load:
Post.eager_load(:comments).where(comments: { approved: true })
```

**Q23. What is `select` in ActiveRecord queries? What gotcha does it introduce?**

```ruby
# select limits which columns are fetched
users = User.select(:id, :name, :email)
# SQL: SELECT id, name, email FROM users

# GOTCHA: accessing non-selected columns returns nil without error
users.first.phone_number  # => nil (not loaded, NO MissingAttributeError in older AR)
# In Rails 7+: raises ActiveModel::MissingAttributeError for strict_loading

# Use case: when you have a wide table and only need a few columns
# Performance: reduces network transfer, memory allocation, object instantiation

# pluck for even lighter: returns arrays, not objects
User.pluck(:id, :email)  # => [[1, "a@b.com"], [2, "c@d.com"]]
# No AR object creation — fastest for bulk data extraction

# pick for single record:
User.where(admin: true).pick(:email)  # => "admin@example.com"
```

**Q24. What is `find_each` and `find_in_batches`? Why are they important for large datasets?**

```ruby
# DANGER with large tables:
User.all.each { |user| user.send_newsletter }
# Loads ALL users into memory at once — potential OutOfMemory

# find_each: processes records in batches of 1000 (configurable)
User.find_each(batch_size: 500) do |user|
  user.send_newsletter
end
# SQL: SELECT * FROM users WHERE id > 0 ORDER BY id LIMIT 500
#      SELECT * FROM users WHERE id > 500 ORDER BY id LIMIT 500
#      ...

# find_in_batches: yields arrays of records
User.find_in_batches(batch_size: 100) do |batch|
  UserMailer.bulk_newsletter(batch).deliver_now
end

# Rails 6+: in_batches returns an ActiveRecord::Batches::BatchEnumerator
User.in_batches.each_record(&:send_newsletter)
User.in_batches(of: 1000).each { |batch| batch.update_all(notified: true) }

# IMPORTANT: find_each forces ORDER BY id ASC — cannot use custom ordering
```

**Q25. What are the differences between `destroy`, `destroy_all`, `delete`, and `delete_all`?**

```ruby
# destroy: loads the record, runs callbacks (before_destroy, after_destroy,
#          after_commit), handles dependent: :destroy associations
user = User.find(1)
user.destroy  # fires callbacks, deletes associated records per :dependent setting

# destroy_all: loads each record, destroys one by one (fires callbacks for each)
User.where(inactive: true).destroy_all  # N SQL DELETE statements + callbacks

# delete: single SQL DELETE, NO callbacks, NO dependent: handling
User.find(1).delete          # DELETE FROM users WHERE id = 1

# delete_all: single SQL DELETE for all matching, NO callbacks
User.where(inactive: true).delete_all  # DELETE FROM users WHERE inactive = true

# Performance table:
# destroy: slow (loads objects, runs callbacks, handles dependents)
# destroy_all: very slow (one-by-one with callbacks)
# delete: fast (one SQL DELETE)
# delete_all: very fast (one SQL DELETE for batch)

# Safety: always use destroy when dependent associations or important callbacks exist
# Use delete_all only for maintenance tasks where callbacks are explicitly not needed
```

**Q29. What is `connection_pool` in ActiveRecord and why does it matter?**

The connection pool maintains a set of reusable database connections. Puma threads borrow connections from the pool, use them, and return them.

```yaml
# config/database.yml
production:
  adapter: postgresql
  pool: <%= ENV.fetch("RAILS_MAX_THREADS") { 5 } %>
  timeout: 5000
  checkout_timeout: 5  # seconds to wait for a connection before raising
```

```ruby
# Pool exhaustion symptom:
# ActiveRecord::ConnectionTimeoutError: could not obtain a database connection
# within 5.000 seconds

# Rule: pool size >= Puma max threads per worker
# If Puma has 5 threads, pool must be >= 5
# With 2 Puma workers: 2 × 5 = 10 connections minimum

# Manual pool inspection:
ActiveRecord::Base.connection_pool.stat
# => { size: 5, connections: 3, busy: 2, dead: 0, idle: 1, waiting: 0, checkout_timeout: 5.0 }
```

**Q30. What is `ActiveRecord::Base.logger` and how do you log all SQL queries?**

```ruby
# In development, SQL is logged to Rails.logger by default
# rails server output shows all queries

# In console — enable query logging:
ActiveRecord::Base.logger = Logger.new(STDOUT)

# In production — use slow query logging:
# PostgreSQL: log_min_duration_statement = 500ms in postgresql.conf

ActiveSupport::Notifications.subscribe("sql.active_record") do |name, start, finish, id, payload|
  duration = (finish - start) * 1000
  if duration > 100  # log queries > 100ms
    Rails.logger.warn "Slow query (#{duration.round(1)}ms): #{payload[:sql]}"
  end
end
```
