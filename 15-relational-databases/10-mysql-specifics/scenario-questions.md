# MySQL Specifics — Scenario Questions

Real-world production situations requiring senior-level MySQL expertise.

---

1. Your MySQL InnoDB database (500GB) has been running for 2 years and a DBA notices that the ibdata1 file (the shared tablespace) is 400GB and growing. Investigating further, the undo logs inside ibdata1 appear to be the cause. A long-running transaction has been open for 6 hours. Explain exactly why this happened, what is preventing undo log truncation, what the immediate remediation is, and how you configure MySQL 8.0 to use separate undo tablespaces to prevent recurrence.

2. Your MySQL replication setup (1 source, 2 replicas, binlog_format=STATEMENT) is experiencing intermittent replication errors. The error logs show "Error executing row event: 'Can't find record in table'" and replication stops. A developer recently added a DELETE statement using a non-deterministic function: `DELETE FROM sessions WHERE RAND() < 0.1 AND expires_at < NOW()`. Explain exactly why this caused replication divergence, what binlog format would prevent this, and what the steps are to resync a replica that has diverged.

3. A high-traffic e-commerce application on MySQL 8.0 reports deadlocks occurring every few minutes on the orders table during peak checkout. The application uses two concurrent transaction patterns: Transaction A (creates order, then updates inventory) and Transaction B (updates inventory, then creates order). Show the exact locking sequence that creates the deadlock, explain InnoDB's deadlock detection mechanism, and describe two application-level solutions that would eliminate the deadlock.

4. Your MySQL database query response time for a critical JOIN query degraded from 50ms to 8 seconds after a data import of 2 million rows. EXPLAIN shows "Using filesort" and "Using temporary" for the first time. No schema changes were made. The query joins three tables with ORDER BY and GROUP BY on different columns. Diagnose the complete set of causes (stale statistics, sort buffer, temp table threshold, index coverage) and provide the tuning plan including specific my.cnf parameters.

5. You are migrating a 1TB MySQL 5.7 database to MySQL 8.0 with zero downtime. The application has been running on MySQL 5.7 for 5 years and uses several deprecated features: GROUP BY implicit sorting behavior, use of YEAR(0) for undefined dates, and utf8 charset (not utf8mb4). Describe the complete migration strategy: compatibility assessment, testing approach, the actual migration steps using replication, cutover procedure, and rollback plan.

6. A startup's MySQL production database shows that their main query — a complex report with 4 JOINs and GROUP BY on a 200M-row events table — takes 45 seconds with FORCE INDEX and 3 minutes without it. The team hardcoded FORCE INDEX in the application code six months ago and forgot about it. Explain: why the optimizer is choosing a worse plan, what the long-term risks of hardcoded FORCE INDEX are (optimizer upgrades, schema changes), and what the correct fix is (optimizer hints vs query rewrite vs index changes).

7. Your MySQL source server has replication to 3 replicas, and one replica has fallen 4 hours behind (replication_lag = 14400 seconds). The source has innodb_flush_log_at_trx_commit=1 and the replica is running with relay_log_recovery=ON but single-threaded SQL thread. The replica disk is an HDD while the source is on NVMe. Diagnose all contributing factors to the lag, explain multi-threaded replication (slave_parallel_type = LOGICAL_CLOCK vs DATABASE), and provide the complete configuration for catching up and maintaining sub-second lag.

8. A security review of your MySQL setup reveals: root user accessible from any host ('root'@'%'), no SSL, binlog format=STATEMENT exposing plaintext passwords in binary logs (due to `INSERT INTO users VALUES ('user', MD5('password'))`), and the application user has SUPER privilege. Prioritize these findings by severity, explain the specific risk each creates, and provide the exact SQL statements and configuration changes to remediate each.

9. Your team needs to implement Change Data Capture (CDC) from MySQL to a Kafka stream for a real-time data pipeline. They are considering Debezium with MySQL binlog consumption. Explain the exact configuration required on the MySQL source (binlog_format, binlog_row_image, GTID settings), the permissions required for the Debezium user, what happens when Debezium falls behind and the binlog rotates, and how GTID replication simplifies the reconnection and position tracking.

10. A SaaS application on MySQL uses a single multi-tenant database with a `tenant_id` column on every table. Performance profiling shows that tenant A (1% of tenants) has 60% of all data and their queries are consuming 80% of database CPU. The engineering team wants to "shard" — moving tenant A to a dedicated MySQL instance. Design the complete sharding architecture using application-level routing, describe the data migration strategy (no downtime), explain how you handle cross-shard queries that the application currently runs, and what monitoring you put in place post-migration.
