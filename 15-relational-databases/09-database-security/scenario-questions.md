# Database Security — Scenario Questions

Real-world security incidents and design challenges at the senior engineer / architect level.

---

1. A security audit reveals that your application's PostgreSQL user has `SUPERUSER` privilege because "the developer needed it to run migrations." The application connects with this credential on every request, handling data for 500,000 users. Describe the immediate remediation plan, the correct privilege model going forward with separate roles for application reads/writes/migrations, and how you implement this with zero downtime migration.

2. A penetration tester submits a report showing they extracted the entire users table by exploiting a search endpoint in your application. The endpoint builds SQL like: `query = "SELECT * FROM products WHERE name LIKE '%" + userInput + "%'"`. The tester used the input `' UNION SELECT username, password_hash, email, NULL FROM users --`. Walk through exactly why this worked, write the secure parameterized version, and describe all the defense-in-depth layers that should have prevented this even if the parameterization was missed.

3. Your company is building a multi-tenant SaaS product where customers' data must be completely isolated from each other. Tenant "Acme" must NEVER be able to see any data from tenant "Globex." You are considering three approaches: (a) Application-layer WHERE tenant_id = $1 on all queries, (b) PostgreSQL Row-Level Security, (c) Per-tenant schemas with separate connection roles. Evaluate each approach against the criteria of: accidental data leakage risk, performance overhead, developer experience, and operational complexity. Which would you recommend and why?

4. Your PostgreSQL database contains PII (Social Security Numbers, credit card numbers) that is currently stored in plaintext TEXT columns. A new compliance requirement mandates that these values be encrypted at rest AND that the application can only access them after re-authentication with a second factor. Design the complete encryption-at-rest solution using PostgreSQL features, including: the encryption library (pgcrypto), key management approach (AWS KMS vs application-level), the SQL for storing and retrieving encrypted values, and how to implement the "second factor before decrypt" requirement at the database layer.

5. An insider threat scenario: a junior DBA who was recently terminated still has credentials to a read-only PostgreSQL role. Your HR system did not automatically trigger a database credential revocation. The DBA knows the connection string and has downloaded a copy of the database schema. What immediate steps do you take to assess the damage and revoke access? Describe the complete off-boarding security checklist for database credentials, including how HashiCorp Vault dynamic credentials would have made this scenario impossible.

6. Your security team requires a complete audit trail of all data access to the `payments` table: who accessed it, when, what query they ran, and what data was returned. The current PostgreSQL setup has no auditing. You cannot modify the application code. Using pg_audit, log_min_duration_statement, and other PostgreSQL features, design the complete audit logging architecture — including what exactly gets logged, where logs are stored, how they are tamper-proofed, and how you implement log retention and review.

7. A developer reports that their application, which correctly uses parameterized queries everywhere, is still vulnerable to SQL injection according to the security scanner. Investigation reveals the application uses a stored procedure that constructs dynamic SQL internally: `EXECUTE 'SELECT * FROM ' || table_name || ' WHERE id = $1' USING user_id`. Explain why this stored procedure is vulnerable despite the parameterized call from the application, write the secure version using format() and EXECUTE with USING, and describe what additional validation you would add.

8. You are designing the database tier for a HIPAA-compliant healthcare application storing patient records. The requirements include: encryption at rest, encryption in transit, access logging, role-based access control with minimum necessary access, and the ability to prove to auditors that a specific user accessed specific patient records. Design the complete PostgreSQL security architecture meeting all requirements, including specific postgresql.conf settings, pg_hba.conf configuration, role structure, RLS policies, and audit logging configuration.

9. Your company's PostgreSQL database is in a private VPC but a network security scan found that the PostgreSQL port (5432) is accessible from 0.0.0.0/0 (the public internet) due to a misconfigured security group. The database has been running this way for 6 months. Describe your complete incident response plan: how you assess what unauthorized access may have occurred, what logs you check (pg_audit, PostgreSQL connection logs, OS auth logs), what immediate remediation steps you take, and how you harden the configuration going forward.

10. A startup is building a financial trading platform and asks you to review their PostgreSQL security model. They have one database role `trading_app` with full SELECT, INSERT, UPDATE, DELETE on all tables, no SSL enforcement, passwords stored as MD5 hashes in pg_shadow, no audit logging, and the database is directly accessible from application servers. The CTO says "security will be added in v2." As the database architect, write your complete security review memo covering: the specific vulnerabilities, the regulatory exposure (PCI DSS, SOX), the minimum viable security improvements for launch, and the medium-term security roadmap.
